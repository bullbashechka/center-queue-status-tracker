# Аудит перебора ID в backend API

Дата проверки: 2026-06-22.

## Вывод

Критичный риск найден в dev API: `POST /api/dev/children/:id/archive` доступен без сессии и использует последовательный `children.id`. Это позволяет перебором ID архивировать активные записи. Этот маршрут нельзя выпускать в продакшен.

Публичных endpoint-ов вида `/api/public/.../:id` не найдено. Публичный статус открывается по `publicToken`, который генерируется как 24 случайных байта в hex.

Admin API закрыт cookie-сессией, но использует последовательные numeric ID для чтения и мутаций. При единой роли сотрудника это не является внешним IDOR, однако остается внутренним риском массового доступа/ошибочных операций и утечки масштаба базы.

## Смежные места без `:id`, влияющие на перебор

| Endpoint / поле | Риск | Оценка |
| --- | --- | --- |
| `GET /api/public/search?iin=...` | Позволяет проверять существование активной заявки по ИИН и получать публичный token. Есть лимит, но он зависит от cookie device ID и IP. | 8/10 |
| Ответ `GET /api/public/status/:token` | Возвращает `id` в публичной схеме `PublicStatusView`. Это не дает доступ по ID, но раскрывает внутренний идентификатор и помогает оценивать размер/динамику базы. | 4/10 |

## План доработок

1. Немедленно убрать dev API из продакшена.
   - Не регистрировать `registerDevRoutes` по умолчанию.
   - Добавить env-флаг вроде `ENABLE_DEV_ROUTES=false`.
   - Если включено локально, закрыть dev routes отдельным секретом или employee session.
   - Добавить тест: без флага `/api/dev/children` и `/api/dev/children/:id/archive` возвращают `404`.

2. Перевести внешние идентификаторы админки на opaque ID.
   - Добавить в `children` отдельное поле `admin_token` или `admin_id` с crypto-random значением.
   - Сделать route-ы `/api/admin/children/:childRef`, где `childRef` валидируется как opaque token.
   - Numeric `id` оставить только внутренним DB primary key.
   - Убрать `id` из публичных ответов.

3. Усилить авторизацию admin API.
   - Явно оформить policy: кто имеет право читать/изменять все записи.
   - Если появятся роли/центры/филиалы, фильтровать каждую операцию по scope сотрудника, а не только по session.
   - Логировать failed access по несуществующим/чужим `childRef`.

4. Нормализовать ответы на отсутствующие ID.
   - Для admin routes валидировать `id`/`childRef` до обращения к БД.
   - Возвращать единый `404` без различий, которые помогают подтверждать существование.

5. Закрыть перебор публичного поиска по ИИН.
   - Не выдавать token сразу после одного фактора знания ИИН: добавить второй фактор из SMS/WhatsApp, одноразовый код или проверку телефона родителя.
   - Усилить rate limit на серверной стороне: доверять `X-Forwarded-For` только от known proxy, добавить глобальный лимит и лимит по IIN-префиксу/хэшу.
   - Сохранять и мониторить превышения/аномалии поиска.

6. Добавить regression-тесты на перебор.
   - Unauthenticated: все `/api/admin/children/:id*` дают `401`, все dev routes выключены.
   - Authenticated: невалидные ID дают `400`, несуществующие дают единый `404`.
   - Public: status не принимает numeric ID, search rate limit нельзя обойти сменой `X-Forwarded-For` при прямом доступе.
