# PRD: Система электронного информирования родителей о продвижении очереди

## О продукте

Веб-приложение для центра, оказывающего услуги детям. Система решает проблему постоянных телефонных звонков родителей с вопросом «когда подойдёт наша очередь». Вместо звонков родитель получает ссылку на персональную страницу, где в реальном времени видит статус своего ребёнка: на каком этапе находится заявка, какой текущий номер в очереди, сколько семей впереди и ориентировочный срок начала занятий.

Продукт состоит из двух частей: внутренней админки для сотрудников центра и публичных страниц статуса для родителей. У родителя есть два равнозначных способа открыть статус ребёнка: персональная ссылка, которую сотрудник может скопировать из админки и отправить вручную, и публичная страница поиска по ИИН (на случай, если ссылка потерялась). Интерфейс и все сообщения — только на русском языке.

---

## Фича 1. Авторизация сотрудников

Доступ в админку имеют только сотрудники центра. Вход осуществляется по логину и паролю. Все сотрудники имеют одинаковые права — разделения на администраторов и обычных пользователей нет, любой авторизованный сотрудник может выполнять любые действия в системе.

Учётные записи сотрудников создаются на этапе настройки системы (без публичной регистрации). После входа сотрудник попадает на главный экран со списком детей в очереди.

---

## Фича 2. Управление детьми в очереди (админка)

Сотрудник вручную заводит ребёнка в систему через админку. Это основной способ попадания ребёнка в очередь — родитель сам заявку не подаёт.

При добавлении ребёнка сотрудник указывает обязательные поля: ФИО ребёнка, ИИН ребёнка (12 цифр) и телефон родителя. Также сотрудник может сразу указать необязательное поле «ориентировочный срок начала занятий» (см. Фичу 5). Других персональных данных система не хранит. ИИН используется как идентификатор для публичного поиска статуса (см. Фичу 8) и должен быть уникальным среди активных записей в системе.

Сотрудник может:

- добавить нового ребёнка;
- отредактировать ФИО, ИИН или телефон;
- удалить ребёнка из активной очереди (например, при отказе семьи);
- скопировать персональную ссылку или готовый текст сообщения для ручной отправки родителю.

В системе есть единый список всех активных детей с возможностью видеть текущий статус и номер в очереди каждого. Список — рабочий инструмент сотрудника для повседневного ведения очереди. В списке есть поиск по ФИО, ИИН и телефону, а также фильтр по статусу.

В момент первого добавления ребёнка система автоматически генерирует для него уникальную постоянную ссылку на страницу статуса и готовит текст приветственного сообщения о постановке в очередь со ссылкой. Сотрудник может скопировать ссылку или весь текст сообщения из админки и отправить родителю вручную через удобный канал.

При удалении ребёнка запись перестаёт быть активной и уходит в архив для сотрудников. Публичная персональная ссылка и поиск по ИИН для удалённой записи больше не показывают статус родителю. Если ребёнка с тем же ИИН нужно снова поставить в очередь после удаления, сотрудник может создать новую активную запись; старая запись остаётся в архиве.

---

## Фича 3. Очередь и позиция в очереди (FIFO)

В центре одна общая очередь на все услуги — без разделения по направлениям, специалистам или типам услуг.

Номер ребёнка в очереди определяется автоматически по порядку добавления: кто раньше попал в систему, тот выше в очереди (принцип FIFO). Сотрудник не управляет порядком вручную.

Когда ребёнок зачисляется (доходит до финального статуса) или удаляется из активной очереди, очередь автоматически пересчитывается, и у всех, кто стоял за ним, номер уменьшается — они продвигаются вперёд.

Для каждого ребёнка система в любой момент знает: текущий номер в очереди и количество семей перед ним.

---

## Фича 4. Статусы ребёнка (этапы)

Каждый ребёнок последовательно проходит четыре этапа:

1. **Документы приняты**
2. **Диагностика пройдена**
3. **Ожидание зачисления**
4. **Зачислен**

Сотрудник вручную переключает статус ребёнка по мере прохождения этапов. Обычный сценарий — движение только вперёд по порядку этапов. Если сотрудник ошибся, он может откатить статус на предыдущий этап или любой другой более ранний этап после подтверждения действия в интерфейсе. Финальный статус «Зачислен» означает, что ребёнок покидает активную очередь, и остальные продвигаются вперёд.

Текущий статус виден сотруднику в админке и родителю на странице статуса.

---

## Фича 5. Ориентировочный срок начала занятий

Для каждого ребёнка сотрудник вручную указывает ориентировочный срок начала занятий — текстовое значение (например, «август–сентябрь 2026 года»). Система не вычисляет этот срок автоматически.

Сотрудник может в любой момент изменить это значение. Срок отображается родителю на странице статуса как ориентир. Поле необязательное: если оно не заполнено, на странице статуса блок со сроком не показывается.

---

## Фича 6. Публичная страница статуса для родителя

У каждого ребёнка есть персональная веб-страница, доступная по уникальной ссылке. Ссылка не требует пароля или входа: родитель открывает её и сразу видит актуальную информацию. Ссылка постоянная — работает для активной записи и после зачисления ребёнка, пока запись не удалена сотрудником.

На странице родитель видит:

- ФИО ребёнка;
- список этапов с отметкой пройденных и текущего (Документы приняты, Диагностика пройдена, Ожидание зачисления, Зачислен);
- текущий номер ребёнка в очереди;
- количество семей перед ним;
- ориентировочный срок начала занятий (если указан сотрудником).

Страница показывает только данные конкретного ребёнка. ФИО других детей и любые персональные данные других семей на странице не отображаются — только обезличенный счётчик «перед вами N семей».

Данные на странице всегда актуальны на момент открытия: при каждом заходе родитель видит текущее состояние очереди и статуса.

Если ребёнок уже зачислен, страница остаётся доступной и показывает финальный статус «Зачислен». Номер в активной очереди и количество семей перед ним для зачисленного ребёнка не отображаются, так как он больше не участвует в очереди.

---

## Фича 7. Заглушка уведомлений и тексты сообщений

В MVP нет обязательной интеграции с WhatsApp Business API, SMS, email или другим внешним провайдером уведомлений. Вместо автоматической внешней отправки система создаёт внутреннюю заглушку уведомлений: готовит русскоязычный текст сообщения, добавляет персональную ссылку на страницу статуса и показывает сообщение сотруднику в админке.

Система создаёт текст уведомления в следующих случаях:

**Постановка в очередь.** При первом добавлении ребёнка система готовит приветственное сообщение о том, что ребёнок поставлен в очередь, со ссылкой на страницу статуса.

**Смена этапа.** При каждом переключении статуса ребёнка (например, с «Документы приняты» на «Диагностика пройдена») система готовит сообщение об изменении.

**Продвижение по очереди.** При изменении номера ребёнка в очереди система готовит сообщение о том, что очередь продвинулась.

Чтобы в будущем не отправлять родителю множество сообщений в день при частом пересчёте очереди, уведомления о продвижении по очереди в MVP также агрегируются на уровне заглушки: по каждому ребёнку хранится не более одного актуального сообщения в день о движении номера. Если за день номер менялся несколько раз, текст обновляется до итоговой актуальной позиции. Итоговое сообщение считается готовым к ручной отправке после 18:00 по часовому поясу центра (`Asia/Almaty`). Ограничение касается только уведомлений о продвижении очереди; сообщения о смене этапа создаются всегда.

Все тексты сообщений содержат ссылку на страницу статуса, чтобы родитель мог посмотреть подробности.

В админке сотрудник видит последний подготовленный текст сообщения и может скопировать его одной кнопкой. Система не отслеживает факт реальной доставки родителю, потому что внешней отправки в MVP нет.

Telegram-бот может быть подключён как опциональный демонстрационный адаптер поверх той же логики сообщений, если для показа продукта нужно отправлять сообщения автоматически в Telegram. В этом случае бот не меняет основные сценарии MVP: система всё равно должна уметь работать без внешнего канала и показывать готовые тексты в админке.

## Фича 8. Публичный поиск статуса по ИИН

Помимо персональной ссылки, у системы есть одна постоянная публичная ссылка, которую центр размещает где угодно: в шапке профиля Instagram, на сайте, в рассылках. Эта ссылка не привязана к конкретному ребёнку — она ведёт на страницу с формой поиска. Способ нужен для родителей, которые не сохранили или потеряли свою персональную ссылку.

На странице поиска родитель вводит ИИН ребёнка и нажимает «Найти». Если ребёнок с таким ИИН есть в системе, открывается та же самая страница статуса, что и по персональной ссылке: этапы, текущий номер в очереди, количество семей впереди и ориентировочный срок. Если ребёнка с таким ИИН в системе нет, показывается понятное сообщение, что заявка не найдена, без каких-либо подробностей.

Поиск не требует пароля или входа. Доступ к статусу контролируется знанием ИИН ребёнка: кто знает ИИН, тот может посмотреть статус. Это осознанный компромисс ради удобства родителей. Чтобы снизить риск подбора чужих ИИН перебором, на форму поиска ставится ограничение: не более 5 попыток за 15 минут с одного IP-адреса/устройства, после чего поиск временно блокируется до окончания периода ограничения.

Страница результата показывает только обезличенные данные конкретного ребёнка (как в Фиче 6): ФИО других семей и любые их персональные данные не отображаются, только счётчик «перед вами N семей».

## Аудит действий сотрудников

Система ведёт внутренний аудит ключевых действий сотрудников. В журнале фиксируются: добавление ребёнка, редактирование данных, удаление из активной очереди, смена статуса, откат статуса и копирование/ручная подготовка сообщения для родителя.

Для каждой записи аудита сохраняются сотрудник, дата и время действия, тип действия и идентификатор ребёнка. Журнал нужен для внутреннего разбора спорных ситуаций и ошибок. Время в журнале отображается по часовому поясу центра (`Asia/Almaty`).

---

## Вне MVP

В первую версию продукта не входят:

- регистрация и личный кабинет родителя;
- онлайн-подача заявки родителем;
- платежи;
- разделение очереди по направлениям, услугам, специалистам или филиалам;
- разные роли и уровни прав сотрудников;
- мультиязычность;
- обязательная интеграция с WhatsApp Business API;
- SMS- и email-уведомления.

## Цветовая палитра и визуальный стиль

Интерфейс выдержан в визуальной айдентике проекта «Аутизм. Мир один для всех» / центров «Асыл Мирас»: спокойные сине-бирюзовые тона, мягкие фоны, много воздуха. Стиль ориентирован на родителей детей с РАС — без агрессивных цветов, резких контрастов и визуального шума. Палитра ниже едина и для админки, и для публичных страниц статуса.

Основные цвета:

- **Основной (бренд), бирюзово-синий** — `#2BA8C9`. Кнопки, ссылки, активные элементы, акценты, шапка.
- **Тёмно-синий (глубокий)** — `#14546A`. Заголовки, основной текст на светлом фоне, иконки.
- **Тёплый акцент (коралл)** — `#F2785C`. Призывы к действию, важные кнопки, выделение ключевого (использовать дозированно).

Цвета статусов (согласованы с иконками на исходных макетах):

- **Пройдено / успех (✅)** — `#3DAE6B` (зелёный). Завершённые этапы: «Документы приняты», «Диагностика пройдена».
- **В процессе / ожидание (🔄)** — `#5B9BD5` (мягкий синий). Текущий этап «Ожидание зачисления».
- **Зачислен (финал)** — `#2BA8C9` (основной бирюзовый). Финальный статус.
- **Предупреждение / не найдено** — `#E0A33E` (тёплый янтарный). Сообщение «заявка не найдена», нейтральные предупреждения.

Нейтральные тона и фоны:

- **Основной текст** — `#2E3A40` (тёмный графит).
- **Второстепенный текст** — `#6B7A82` (серо-голубой).
- **Фон страницы** — `#F5FAFB` (очень светлый бело-голубой).
- **Карточки и блоки** — `#FFFFFF` (белый).
- **Границы и разделители** — `#E2EBEE` (светло-серо-голубой).

Все сочетания текста и фона должны проходить по контрастности (минимум AA по WCAG), чтобы интерфейс был читаемым в том числе для родителей, открывающих страницу с телефона на улице.
