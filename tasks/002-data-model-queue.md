# 002. Модель данных ребёнка, очереди и архива

## Этап

Фундамент продукта.

## Цель

Создать модель данных для сотрудников, детей, активной FIFO-очереди, архива, персональных ссылок, статусов, событий уведомлений и аудита.

## Контекст из PRD

Система хранит ФИО ребёнка, ИИН, телефон родителя и необязательный ориентировочный срок начала занятий. ИИН уникален среди активных записей. Удалённые записи уходят в архив.

## Scope

- Спроектировать и реализовать таблицы/сущности.
- Добавить статусы: «Документы приняты», «Диагностика пройдена», «Ожидание зачисления», «Зачислен».
- Добавить признак активности/архива.
- Добавить постоянный публичный токен персональной ссылки.
- Подготовить поля для позиции FIFO и расчёта количества семей впереди.

## Out of Scope

- UI админки.
- Реальная отправка через внешние каналы уведомлений.
- Отображение публичной страницы.

## Требования

- Активный ИИН должен быть уникальным.
- Архивная запись не должна блокировать создание новой активной записи с тем же ИИН.
- Персональный токен ссылки должен быть уникальным и непредсказуемым.
- Финальный статус «Зачислен» исключает ребёнка из активной очереди, но не удаляет запись.

## Acceptance Criteria

- Можно создать активную запись ребёнка с обязательными полями.
- Нельзя создать две активные записи с одним ИИН.
- Можно архивировать запись и после этого создать новую активную запись с тем же ИИН.
- Для активных незачисленных детей можно вычислить номер в очереди и количество семей перед ними.
- Для зачисленного ребёнка номер активной очереди не возвращается.

## Зависимости

Задача 001.

## Implementation Notes

FIFO-порядок должен опираться на стабильное время/порядок добавления, а не на редактируемые поля.
