# 012. Проверка MVP-сценариев и подготовка к выпуску

## Этап

UI, качество и выпуск.

## Цель

Проверить, что весь MVP из PRD работает end-to-end, а явно исключённые функции не попали в объём первой версии.

## Контекст из PRD

MVP включает админку, публичные страницы, поиск по ИИН, FIFO-очередь, заглушку уведомлений и аудит. Вне MVP: родительская регистрация, онлайн-заявки, платежи, направления услуг, роли сотрудников, мультиязычность, обязательная интеграция с WhatsApp Business API, SMS и email.

## Scope

- End-to-end проверка основных сценариев.
- Проверка edge cases из PRD.
- Проверка недоступности удалённых записей.
- Проверка зачисленного ребёнка на публичной странице.
- Проверка подготовки и копирования текстов сообщений.
- Проверка rate limit поиска по ИИН.
- Подготовка минимальной инструкции запуска/эксплуатации.

## Out of Scope

- Расширение продукта за пределы PRD.
- Нагрузочное тестирование промышленного уровня, если оно не требуется для MVP.

## Требования

- Все критические сценарии должны быть проверяемыми вручную или автоматизированными тестами.
- Нельзя добавлять функции из раздела «Вне MVP» как обязательные.
- Ошибки не должны раскрывать персональные данные.

## Acceptance Criteria

- Сотрудник может пройти сценарий: вход, добавление ребёнка, просмотр в списке, смена статуса, зачисление.
- Родитель может открыть персональную ссылку и увидеть актуальные данные.
- Родитель может найти ребёнка по ИИН и попасть на ту же страницу статуса.
- Удалённая запись недоступна родителю через ссылку и поиск.
- Зачисленный ребёнок показывает финальный статус без номера активной очереди.
- Сотрудник может скопировать ссылку и готовый текст сообщения для родителя.
- Аудит содержит записи по ключевым действиям.
- Есть понятная инструкция по переменным окружения, запуску и базовой настройке сотрудников.

## Зависимости

Все предыдущие задачи.

## Implementation Notes

Финальная проверка должна идти по пользовательским сценариям, а не только по отдельным компонентам.
